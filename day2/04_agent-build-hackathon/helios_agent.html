The demo in 60 seconds

Helios Security needs to answer RFPs in minutes, not hours, without contradictory answers across questions. We built a 5-agent team that parses an unseen RFP, retrieves from a small knowledge base, drafts cited answers, validates them as a set, and revises any flagged by the validator. The whole pipeline runs in the browser against the Anthropic API. Open the Run tab, paste an RFP (or load a sample), and watch the cards and live-trace fill in.

Why this shape

  • Haiku for routing, Sonnet for judgment. Parser + Retriever are chatty and low-stakes — Haiku gives ~4× cost cut without quality loss. Drafter, Validator, Reviser need the bigger model.
  • Validator as a separate agent. The brief calls cross-answer consistency the #1 failure mode. A drafter looking at one question at a time can't catch contradictions; a validator with the full set does.
  • Bounded revision + recheck. 1 revision pass is enough in practice and avoids spin. If the contradiction survives, we ship with a flag rather than loop forever.
  • Pure browser. Zero backend. Paste API key, paste RFP, run. Same agent contract is mirrored in the Jupyter notebook for anyone who wants to extend the KB or run evals locally.

How to use this UI

A quick tour. The left rail drives runs, the center shows results, the right pane streams the live agent trace, and the bottom chat answers questions about the run, the architecture, or the KB. All borders are draggable — grab the violet seam between any two panes to resize.

Left rail — Controls

  • Anthropic API key. Auto-fills from your env if you launched via serve.py, otherwise from the browser cache, otherwise paste it. The key never leaves your machine except as an x-api-key header to api.anthropic.com.
  • Models. Drafter / Validator / Reviser run on Sonnet; Parser / Retriever run on Haiku. Defaults are the latest IDs but you can swap.
  • Run controls. Max revisions caps the Reviser loop (default 1). Mode: parallel fires all questions at once, sequential serialises them — try sequential if you're hitting rate limits.
  • RFP input. Pick a baked-in sample (5 shapes: table, list, prose, edge-cases, adversarial) or paste your own RFP text. Anything reasonably formatted works — the parser falls back to blank-line chunking if it can't structure it.
  • RUN AGENT TEAM. Kicks the pipeline. Cards populate left-to-right through stages.

Center — Run tab

  • Scoreboard tiles. Questions parsed, OK, flagged, failed, revisions, total duration.
  • Cards. One per question. State colors: violet=in-progress, green=ok, amber=flagged, red=failed. Each card shows category badge, answer preview, source count, confidence pill, and any flags (look for kb-gap and unresolved-contradiction).
  • Export JSON. Downloads the full structured response — questions, answers, sources, validation report, metadata.

Right pane — Live trace

Append-only timestamped log of every agent turn. Each entry shows time, role (parser / retriever / drafter / validator / reviser), target question (if applicable), outcome summary, and latency. Errors are red, warnings amber.

Bottom — Follow-up chat

  • Enabled at all times, not just after a run. Ask anything: explain the architecture, walk me through Q3's sources, why is Q2 flagged, what would happen if KB had no compliance entry.
  • The chat receives a RUN_SNAPSHOT JSON every turn containing the current RFP, all cards, the trace tail, the full final-answer set, validation, and a KB summary — so it grounds on your actual run, not just architecture knowledge.
  • Multi-turn — the assistant remembers the last ~12 turns of conversation.

Tips for the surprise RFP

  • If the parser tags everything as other, your RFP shape is unusual — that's expected, the Drafter still answers, just with weaker category routing.
  • kb-gap on a card means the Retriever found nothing — the answer is honestly hedged, not fabricated. That's the right behavior, not a bug.
  • unresolved-contradiction means the Reviser tried once and the Validator still complained. Open the JSON export to see the diff.
